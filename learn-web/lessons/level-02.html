Variables & Types

Store data in named boxes the computer can remember.

Warm-up question

"If I asked you to remember my age, where would you write it down so you can look it up later?"

A variable is exactly that "place where you write it down."

1 The concept

A variable is a named storage spot for a value. You declare it once, then use the name as a stand-in for the value everywhere else.

Memory Palace — Station 2: The Locker Room. Walk past the PA speaker from Station 1 and through the swinging door into the locker room. Rows of metal lockers, each with a label slot at the top: age, name, isStudent, price. You can open a locker, put exactly one item inside, and the label stays the same. Hear the clang of the metal door.

Bind it: the label = the variable name. The item inside = the value. The size and shape of the locker = the type — a tall narrow locker only fits a number (int), a wide one fits a long sentence (string), a tiny one fits one toggle switch (bool). Try to shove a shoe into the toggle-switch locker and the gym manager (the compiler) stops you at the door. 
int age = 25;
string name = "Sam";
bool isStudent = true;
double price = 19.99;

Console.WriteLine($"Hello {name}, you are {age}.");

Read the shape: type name = value; — that pattern never changes.

Drill the pattern

Cover the screen and ask them to say the pattern out loud: "type, name, equals, value, semicolon." Make them repeat it 3 times. It's the rhythm they'll use for thousands of lines of code.

Ask

"In int age = 25; which part is the name, and which part is the value?"

2 The four core types

  • int — whole numbers like 0, 42, -7. No decimals.
  • double — numbers with decimals: 3.14, -0.5, 19.99.
  • string — text in double quotes: "hello".
  • bool — true or false. Just those two.
Watch out: int age = 25.5; will not compile. 25.5 is a double, and C# refuses to silently lose the .5. Strong typing protects you.
String interpolation: A string starting with $ lets you drop variables in with { } braces.
$"Hello {name}" → "Hello Sam".
Interview answer: "C# is statically and strongly typed: every variable has a type known at compile time, and the compiler rejects assignments between incompatible types unless you explicitly cast. This catches bugs before runtime."

Hammer the difference: int vs double

Have the student type int x = 25.5; in the editor on purpose. Show them the red squiggle. Read the error message together. Reading compiler errors is a skill, not a punishment.

Common mistakes

  • Forgetting double quotes around a string → "name does not exist in the current context."
  • Using single quotes for a string → "Too many characters in character literal."
  • Storing a number in a string variable — works but means math won't work later. Worth flagging.

3 Strong typing, compiled & interpreted

What is strong typing?

Strong typing means every variable has a fixed type, and the language enforces the rules about that type. If you declare a variable as int, it can only ever hold whole numbers. The language will never secretly convert it to something else behind your back.

Analogy — the labelled box: imagine you have a box labelled "whole numbers only". A strongly typed language is the strict rule that says you cannot put a letter or a decimal in that box — not now, not ever. A weakly typed language would let you sneak a decimal in and silently round it, which causes bugs that are very hard to find.

C# is statically and strongly typed:

  • Statically — the type of every variable is decided at compile time (before the program runs).
  • Strongly — the compiler refuses to silently convert between incompatible types. You must be explicit.
int score = 100;
      score = "hello";   // ❌ compiler error — cannot assign string to int

What does "compiled" mean?

When you write C# code, the computer cannot run it directly — it is just text. Compiling is the step that translates your human-readable code into instructions the computer can execute. Think of it as a translator finishing the whole book before anyone reads it.

Analogy — the book translator: you write a novel in English. A compiler is a translator who reads the entire novel first, checks for grammar errors, and produces a finished version in the target language. Only after the full translation is complete does the reader (the computer) get the book. If there is an error on page 47, the translator stops and tells you before any reader ever sees a single word.

In .NET, C# is compiled to an intermediate language called IL (Intermediate Language). When you run the program, the .NET runtime uses a JIT (Just-In-Time) compiler to turn IL into the actual machine code for your CPU. You do not need to memorise this — just know that compilation happens before runtime, so errors are caught early. 

// You type this (C# source code)
      int age = 25;
      
      // dotnet build compiles it
      // → IL code (inside the .dll file)
      // → machine code at runtime (JIT)

What is an interpreter?

An interpreter is the opposite approach: instead of translating the whole program first, it reads and executes your code one line at a time, right now, while the program is running.

Analogy — the live interpreter: now imagine a live human interpreter at a conference who whispers a translation into your ear, sentence by sentence, as the speaker talks. Fast to start, but any grammar mistake only becomes obvious the moment that sentence is spoken — possibly in the middle of your live presentation.

Examples of interpreted languages: Python, JavaScript (in the browser), Ruby.

Compiled (C#, C++) Interpreted (Python, JS)
When is it translated? Before the program runs While the program runs
When are errors found? At compile time (before running) At runtime (while running)
Speed Usually faster Usually slower
Type checking Static — checked before running Dynamic — checked while running
Why does this matter for you? When you see a red squiggle in VS Code before you even run your program, that is the compiler catching a type error. In Python or JavaScript, that same mistake would only blow up at runtime — possibly after you ship to users.
Interview answer: "C# is a compiled, statically typed language. All type errors are caught at compile time, before the program runs. An interpreter would execute code line by line at runtime, which means type errors appear later. C#'s compiled model makes bugs cheaper to find and fix."

How to explain this live

Open Variables.cs and intentionally type int x = "hello";. Point to the red squiggle before pressing F5. Say: "That red line is the compiler. The program hasn't started yet. The mistake is already found."

Then switch to a Python REPL and assign a number to a string variable. Show that Python only complains when that line actually runs. Ask: "Which would you rather have?"

Key phrase to have them repeat

"Strong typing means the compiler knows the type of every variable before the program runs, and will not let me mix them up."

4 var — let the compiler decide

You can write var instead of the type. The compiler picks the right type from the value on the right:

var age = 25;          // int
var name = "Sam";       // string
var isOk = true;        // bool

It is still strongly typed — var is just shorter syntax. age = "hi"; after this would still fail.

When to introduce var

For the first 2–3 levels, ask them to write the explicit type (int, string). Seeing the type forces them to think about it. var is a great shortcut once they think in types, not before.
